## D-1 ✅ Моделі-відповідачі = поточна реальність

**Рішення:** канон — **Groq (`llama-3.3-70b-versatile`) + Cerebras + SambaNova** як відповідачі, **Gemini (`gemini-2.5-flash-lite`) — суддя**. GPT/Claude з первісного концепту — **опційні майбутні провайдери**, не наявні.

## D-2 ✅ Зараз — один користувач; пізніше — *мінімальні* акаунти

**Рішення:** поточний режим — однокористувацький. У майбутньому — **мінімальні акаунти/логін** (легкі, без enterprise-auth): логін відмикає збереження чатів і синхронізацію персоналізації.
↪ Код залишає місце під ключ користувача; повноцінний auth не будуємо до відповідної фази ([09](09-roadmap.md), Фаза 5).

## D-3 ✅ Single — ефемерний; збереження — окрема майбутня фіча

**Рішення:** одинарний чат (Single) **не зберігається як іменований чат** (підтверджено власником). «Збереження чатів (до 3)» — окрема фіча, прив'язана до акаунтів, і стосується збережених (Compare) чатів, а не Single.
↪ Поточний дефект (Single пише в `ChatBuffer`) **виправити** у Фазі 0 ([09](09-roadmap.md)). Збережені чати — Фаза 4.

**Уточнення (PH13, 2026-05-29):** «ефемерність» Single тепер **візуальна, не на рівні БД**. Ходи Single **записуються в rolling-історію** (`interactions`) після стріму — для персоналізації; це **не** створює збережений (Compare) чат. UI має кнопку «Очистити», що прибирає лише **візуальний** тред (дані в БД лишаються). Збережені чати лишаються **виключно** для Compare; у Single секції збережених чатів немає.

## D-4 ✅ RAG / БД / Деплой — ПІДТВЕРДЖЕНО як майбутній обсяг

↪ Внесено як зобов'язані фази в [09-roadmap.md](09-roadmap.md). Принцип: не починати RAG поверх ядра з боргом.

## D-9 ✅ Суддя — на Groq (Gemini лишається для embeddings)

**Рішення (PH13, 2026-05-29):** перенести **AI-суддю** з Gemini на **Groq**, нейтральною моделлю `qwen/qwen3-32b`. Причина: безкоштовний тариф Gemini обмежує `generate_content` до **20 запитів/добу** на `gemini-2.5-flash-lite` — суддя постійно падав у rule-based fallback. Groq дає ~60/хв і ~14 400/добу. Це **уточнює D-1** (де суддею був Gemini).
↪ `SELECTOR_PROVIDER=groq`, `SELECTOR_MODEL=qwen/qwen3-32b` (`config/selector_config.py`); суддя будується окремо від відповідачів (`ProviderService.build_judge`), тож **немає self-bias** (qwen ≠ жоден з відповідачів groq/cerebras/sambanova). `SELECTOR_MAX_TOKENS=2048` — запас для reasoning + JSON. **Gemini лишається для RAG-embeddings** (окрема, набагато більша квота). Ручний перевибір **оборотний** — можна погодитись із вибором судді назад.

## D-10 ✅ Адмінпанель + ліміти акаунтів + реєстрація за кодом (план 015)

**Рішення (2026-05-29):** власник дає доступ працедавцям і хоче спостерігати/обмежувати їхні запити, щоб не вичерпати безкоштовні API-ключі.

- **Модель ліміту = кількість запитів.** Compare (мульти) рахується як **1 запит**, Single = 1 запит.
- **Per-user ліміти у БД:** `max_requests_per_minute`, `max_requests_per_day`. Дефолт працедавцям: **5/хв, 30/день**. **Адмін: null/null = безліміт.**
- **Безліміт (null) → банер обмеження НЕ показується** і enforcement не застосовується.
- **Адмін** — прапорець `is_admin` (доступ до адмінпанелі + безліміт). Адмін-акаунт визначається через `ADMIN_USERNAME` (.env).
- **Реєстрація лише за кодом** `REGISTRATION_CODE` (.env): без валідного коду — `403 invalid_registration_code`. Працедавцям акаунти створює адмін у панелі (там же ліміти).
- **Аудит:** окрема append-only таблиця `usage_events` (не обрізається, на відміну від rolling `interactions`) — для лічби лімітів і перегляду в адмінці; зберігає витрачені токени.
↪ Реалізація — [plans/015-admin-and-quotas.md](plans/015-admin-and-quotas.md). Контракти → `03-api-contracts.md` (admin, register code, `429 quota_exceeded`), `04-data-models.md` (`usage_events`, нові поля `User`).
↪ **Реалізовано (PH15, 2026-05-30):** усі пункти вище в коді й покриті тестами (BE `test_admin_quotas.py`; FE адмінпанель/банер/реєстрація). Деталі — [08-current-state.md](08-current-state.md) (секція PH15). Гейти зелені: BE 101 passed + ruff/black; FE tsc/eslint/prettier/vitest(12)/build.

## D-15 ✅ BYOK: життєвий цикл доданої моделі (план 020)

**Рішення (2026-05-31):** як поводяться **додані (custom) моделі** у вікні «свої ключі». Продовжує D-14.

- **Зберігається лише валідна+робоча додана модель.** Порожні та невалідні custom-рядки **не** потрапляють у `sessionStorage`.
