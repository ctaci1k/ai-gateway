## Розділ 1 — Single (одинарний чат)

**Призначення:** прямий діалог з однією обраною моделлю.

**Поведінка:**
- Користувач обирає провайдера (за замовчуванням `groq`).
- Відповідь приходить **стрімінгом** (token-by-token).
- Історія цього чату **не зберігається** — ні на клієнті між сесіями, ні (за цільовим задумом) на сервері.

> ⚠️ **Конфлікт реалізації:** наразі бекенд **усе одно записує** одинарний чат у пам'ять (`/chat/stream` → `ChatBuffer.add_message`). Це суперечить задуму «не зберігається». Рішення — у [10-open-decisions.md](10-open-decisions.md) (D-3).

**User flow:**
1. Користувач вводить запит → Send.
2. Бекенд стрімить відповідь однієї моделі.
3. Відповідь з'являється поступово в області чату.

## Розділ 2 — Compare (порівняння + вибір) — ГОЛОВНИЙ

**Призначення:** отримати кілька відповідей на один запит і обрати найкращу.

**Поведінка:**
- Один запит надсилається **паралельно** у кілька моделей (зараз: `groq`, `cerebras`, `sambanova`).
- Окремий **AI-суддя (Gemini)** аналізує всі відповіді й обирає найкращу (`best_response`).
- Якщо суддя недоступний / повертає невалідний результат — спрацьовує **rule-based fallback selector** (детермінований скрипт-оцінювач).
- Користувач бачить найкращу відповідь, а через **Show All Responses** — усі відповіді в колонках.
- Користувач може **вручну обрати іншу модель** («Обрати цю відповідь»). Цей вибір **навчає** систему персоналізації.

**Критерії суддівства** (оцінює Gemini, fallback дублює правилами):
- коректність / повнота;
- структурованість;
- читабельність;
- якість пояснення / стиль;
- відповідність уподобанням користувача (персоналізація).

**User flow:**
1. Користувач вводить запит → Send (у режимі Compare).
2. Бекенд паралельно опитує всі моделі, збирає `all_responses` + метадані виконання.
3. Суддя обирає `best_response` + `scores` + `reason` + `confidence`.
4. UI показує найкращу відповідь і (за бажанням) усі колонки з оцінками.
5. (Опційно) Користувач натискає «Обрати цю відповідь» на іншій моделі → надсилається сигнал персоналізації.

---

## Персоналізація (User Preferences)

Система накопичує профіль користувача й передає його судді як контекст:

- які моделі користувач обирає (`preferred_models`);
- ручні вибори (`manual_model_selections`);
- згоди/незгоди з суддею (`selector_agreements` / `selector_disagreements`);
- лічильники використання (selector, compare, total).

Суддя враховує цей профіль при наступних виборах → вибір стає персоналізованим.

---

## Сповіщення / fallback-банери

- Якщо Gemini-суддя недоступний — користувач **повинен це бачити**:
  банер на кшталт *«Gemini Judge недоступний. Використано Fallback Selector.»*
- Резервні спливаючі повідомлення про тимчасову недоступність моделі — вибір тоді робить детермінований скрипт (fallback selector).

---

## Сховище чатів (тестова версія)

- **До 3 чатів** одночасно.
- Перемикання / створення / видалення чатів (UI).
- 🔮 У майбутньому: логін і збереження між сесіями (див. [09-roadmap.md](09-roadmap.md)).

---

## Локалізація

- Інтерфейс підтримує **3 мови**: українська (`uk`), польська (`pl`), англійська (`en`).
- У UI заборонено хардкодити текст — лише через `t("key")`.

---

## Межі MVP vs майбутнє

| Можливість | MVP (зараз/найближче) | Майбутнє |
|---|---|---|
| Single chat (стрім) | ✅ | — |
| Compare + Gemini selector + fallback | ✅ | — |
| Персоналізація (in-memory) | 🟡 | Персист у БД |
| Сховище до 3 чатів | 🟡 UI | Персист + логін |
| Локалізація (3 мови) | ✅ | + мови |
| RAG / завантаження PDF | ⛔ | 🔮 |
| Мульти-юзер / акаунти / сесії | ⛔ | 🔮 |
| Persistent storage (БД) | ⛔ | 🔮 |

> RAG, акаунти, збереження чатів, логування, БД, Docker/VPS — **підтверджений майбутній обсяг** (не MVP, але зобов'язані фази), а не «можливо». Послідовність і прив'язку до днів 10–14 див. у [09-roadmap.md](09-roadmap.md); ухвалені рішення — у [10-open-decisions.md](10-open-decisions.md).
