## 0. Стандартна авторизація (важливо)

Власник дав **постійний дозвіл** на безперервне виконання всієї програми. Тому:

- **НЕ** питай «чи продовжувати», «чи почати наступну фазу», «чи все гаразд» — просто продовжуй.
- **НЕ** зупиняйся після завершення фази — одразу переходь до наступної за порядком.
- Працюй повноцінно: **без заглушок, TODO-«потім», напівфіч, закоментованого коду**.
- **Нульовий технічний борг / без костилів:** усе робиться одразу повноцінно й до кінця; помічений борг у межах фази — усувай, а не відкладай. Кожен крок лишає робочий, повністю функціональний результат.
- Зупинка дозволена **лише** за умов із розділу 4.

## 3. Цикл виконання (на кожен крок)

1. Виконай крок **повністю** (за описом у під-плані; контракти — з `docs/03`, `docs/04`).
2. **Перевір** результат (build/тест/ручна перевірка, що доречно).
3. Постав `[x]` біля кроку в під-плані.
4. Онови **GLOBAL STATE** (вгорі цього файлу): активний/останній крок, `Оновлено`.
5. На межі фази — онови «СТАН» під-плану; коли всі кроки + DoD виконані → постав `status: done` у під-плані й познач фазу done у GLOBAL STATE.
6. **Одразу** переходь до наступного кроку/фази. Без пауз.

## 4. Коли ЗУПИНИТИСЬ (тільки це)

- **Справжній блокер:** неоднозначність/суперечність, де будь-який вибір ризикує зробити НЕ те. Дія: запиши в GLOBAL STATE → «Блокери», і:
  - якщо є безпечний задокументований дефолт (`docs/10`) — застосуй його й продовж;
  - якщо ні — **пропусти** крок до наступного незалежного й познач пропуск; не вгадуй мовчки.
- **STOP-правило:** вимога суперечить ухваленим рішенням (`docs/10`)/здоровому глузду/патернам → зафіксуй конфлікт у «Блокери» й обери найбезпечніший варіант або пропусти.
- **Небезпечна/незворотна дія поза обсягом** (видалення даних, push, зовнішні виклики) → не роби, зафіксуй.
- **Втручання власника.**
- ❌ Завершення фази — **НЕ** причина зупинятись.

## 5. Планка якості (для тебе, бо ти без контексту чату)

- **Не вигадуй стан** — звіряй із кодом / `docs/08`. Контракти — з `docs/03`, `docs/04`.
- **Frontend golden rules:** без дублювання JSX/стилів/кольорів/текстів; без `fetch` у компонентах (лише `services/`); тексти лише через `t("key")` (uk/pl/en); кольори/відступи лише через design tokens (`theme/`); стан — через `store/`.
- **Уникай граблів:** дублювання, race conditions, необроблені помилки, відсутні стани loading/empty/error, витоки секретів, n+1, a11y, магічні константи. Бачиш у коді — фіксуй у межах відповідної фази.
- **Без регресій:** режими Single/Compare мусять працювати після кожної фази.
- **Прогрес — лише у файлах** (GLOBAL STATE + під-плани), ніколи «в голові»/чаті.
- **Git:** коміть/пуш **лише** якщо власник прямо попросив. Інакше зміни лишаються в робочому дереві (цього досить для відновлення).

## 8. Паралелізм через субагентів (обов'язково, де доречно)

Де кроки **незалежні** (різні файли, без спільного порядку) — **розпаралель** через субагентів: **по 3–4 одночасно** (розумний максимум, не більше). Приклади:
- Міграція багатьох компонентів на токени+`t()` (PH5) — група компонентів на субагента.
- True-async переписування кількох провайдерів (PH2) — по провайдеру.
- Словники i18n uk/pl/en (PH5) — по локалі.
- Тести по модулях (PH1/PH4/PH8…) — по модулю.
- Незалежні фічеві UI-екрани (PH8–PH10) — паралельно.

Правила:
- Кожен субагент отримує **самодостатнє** завдання + ці правила (розділ 5) + точний перелік файлів; повертає короткий звіт/діфф.
- **НЕ** паралель на одному файлі; **НЕ** паралель залежних кроків.
- **Орхестратор ЗАВЖДИ верифікує** вихід субагентів (build/lint/контракти) — без сліпої довіри; після батчу — build/тести → онови STATE → далі.
- Ліміт одночасності ~4. Паралелізм — лише **всередині** кроку/фази; порядок фаз зберігається.
