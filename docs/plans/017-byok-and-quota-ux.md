- **(B) Картка недоступної моделі.** Показувати **реальну display-назву** моделі (динамічно з реєстру/override, не сирий ключ; якщо впаде Llama — покаже Llama). Візуально: лишити пунктир, але **залити фон** легким відтінком (як у вибраної фіолетової картки) і зробити текст читабельним; бейдж не «ріже» лінію. Мінімум змін.

### Ліміти (живі, динамічні)
- Банер лімітів — **два окремі рядки**: «хвилина» і «день». Значення — з БД цього акаунта.
  - Якщо ліміт на хвилину = `null` (безліміт) → рядок хвилини **не показується**; якщо день = `null` → рядок дня не показується. Якщо обидва `null` → банера нема.
- **Хвилина (NQ2):** вікно **відкривається першим запитом** і триває рівно 60с; по завершенні — **повний скид** (усі слоти знову доступні). Рядок: `лишок/ліміт` + **відлік до скиду** (формат `mm:ss` або просто секунди — обрати акуратний/професійний). Поки запитів нема — повний ліміт, без таймера.
  - Рекомендована реалізація без міграцій: `window_start = найраніший usage_event за останні 60с`; `used_minute = к-ть подій від window_start`; `remaining_minute = max(0, limit − used_minute)`; `minute_resets_in = max(0, 60 − (now − window_start))`; якщо за 60с подій нема → `remaining = limit`, таймера нема. **Enforcement використовує те саме вікно** (без розбіжності з UI).
- **День (NQ3):** **календарний день за Europe/Warsaw**, скид о **00:00 польського часу** (DST-aware, `zoneinfo`). Рядок: `лишок/ліміт` **без таймера**; додати підпис «Скидається о 00:00 за польським часом». `day_start = сьогодні 00:00 Europe/Warsaw → UTC`; `used_today = події від day_start`; enforcement — те саме вікно.
- **Жива індикація:** після кожного запиту — `AuthContext.refresh()`; для хвилинного рядка — клієнтський тикер 1с від `minute_resets_in`; коли дійшло до 0 — повторний `refresh()`. День оновлюється лише числом (без таймера).
- **`/auth/me` розширити:** додати `remaining_this_minute`, `minute_resets_in_seconds` (і за потреби `day_resets_at`). `null`, коли відповідний ліміт безлімітний.
- При активних BYOK-ключах текст банера уточнюємо: ліміти стосуються **вбудованих** ключів акаунта.

### BYOK — свої API-ключі (головна фіча)
- **Дві точки входу** у вікно налаштувань (відкривають одне й те саме спливаюче вікно):
  1) кнопка **над `AuthorCard`** («Використання своїх ключів»);
  2) **«+» квадратик** у перемикачі моделей Single (після наявних чіпів).
- **Доступно всім типам акаунтів** (адмін / лімітний / безлімітний).
- **Поля (NQ1, NQ4):**
  - **1-й рядок — Суддя**: підпис «Суддя», показано наш дефолт-нейм (Qwen); користувач вписує `api_key + model_id`.
  - **3 рядки відповідачів** (Llama / GLM / DeepSeek-слоти): `api_key + model_id`.
  - **Додавання** ще відповідачів — **до 5 усього**; доданим потрібен `base_url` (OpenAI-сумісний) + `api_key + model_id`. Дефолтним слотам `base_url` зафіксовано endpoint'ом провайдера.
- **Безпека/зберігання (NQ3, NQ5):**
  - **Не зберігаємо в БД і не логуємо.** Ключі живуть у **sessionStorage** браузера (зникають при закритті вкладки/браузера), передаються **в кожному запиті** по захищеному каналу; бекенд використовує **транзитно** (in-memory на час запиту) і відкидає. У проді — обов'язково HTTPS (зафіксувати в `DEPLOY.md`).
  - Інпути ключів — **масковані зірочками**, з «оком» показати/сховати; при відкритті вікна **завжди приховані** (навіть раніше введені). Активні ключі **лишаються у формі й редагуються**.
  - **Валідація на «Зберегти»:** спінер → тест-виклик кожного заповненого `(base_url+key+model)` → робочі активуються; неробочі **лишаються у формі**, інпут підсвічується **червоним** + конкретне повідомлення «цей ключ не працює» (не видаляємо — можливо одрук в 1 символ). Окремий бекенд-ендпоінт валідації (без збереження).
- **Назва моделі (NQ4):** у UI показуємо **рівно введений `model_id`** (правдиво), скрізь: чіпи Single, колонки Compare, картка недоступної, банер судді.
- **Single (NQ6):**
  - Додані/перевизначені моделі — **обираються** як чіпи; «+» відкриває налаштування.
  - **Модель, вписана в поле Судді, теж доступна для вибору в Single** (щоб не обмежувати користувача); у шапці вікна налаштувань додати інфо про це.
  - Якщо обрана модель на **своєму** ключі → над банером лімітів **зелена плашка** «ваша модель … підключена, без обмежень».
- **Compare:**
  - Сітка **адаптивна** під 4–5 відповідачів.
  - **Червона** плашка при кліку на Compare, якщо **не всі** учасники (всі відповідачі + суддя) на своїх ключах: «доступно з обмеженнями…».
  - Повний безліміт Compare — лише коли **всі** учасники на своїх ключах.
- **Квоти з BYOK (Q6/Q7):**
  - **Single:** списується, лише якщо обрана модель — на **дефолтному** ключі; на своєму — **не списується**.
  - **Compare:** списується **1 запит**, **крім** випадку, коли **всі** учасники (відповідачі + суддя) на своїх ключах → безліміт.
  - **Адмін (Q8):** жодних лімітів узагалі; але блок BYOK у нього **є**, як у всіх.
  - Інфо-тексти у вікні: «свій ключ → без списання»; «якщо не всі 4 (суддя + мін. 3) — доданою моделлю безплатно лише в Single і лише якщо це не суддя»; модель із поля судді можна юзати в Single.
- Якщо зі свого ключа нема відповіді → та сама картка «тимчасово недоступно / ліміт» (Bug B).
